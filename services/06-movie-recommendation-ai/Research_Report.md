# Research Report — Movie Recommendation AI

MovieLens 100K 데이터를 활용한 영화 추천 AI 서비스의 데이터 분석, Feature Engineering, 추천
알고리즘 비교 및 최종 모델 선정 과정을 정리한 문서입니다. 모든 수치는
`ai-server/fe/movie_recommendation_analysis.ipynb`를 실제로 실행한 결과를 기준으로 작성했습니다.

---

## 11.1 프로젝트 개요

### 배경

온라인 영화 서비스에서 사용자는 종종 "지금 이 영화와 비슷한 다른 영화가 뭐가 있을까?"라는 질문에
답을 얻고 싶어 합니다. 로그인이나 긴 시청 이력 없이도, **영화 한 편을 선택하는 것만으로** 유사한
작품을 안내받을 수 있다면 콘텐츠 탐색 경험을 크게 개선할 수 있습니다.

### 목적

MovieLens 100K 데이터셋(사용자 943명, 영화 1,682편, 평점 100,000건)을 사용해, 사용자가 선택한
영화와 유사한 영화를 추천하는 AI 서비스를 구축합니다.

### 해결하려는 문제

- 평균 평점만으로 영화를 추천하면, 평가자가 극소수인 영화가 왜곡된 순위를 차지하는 문제가 발생합니다.
- 콘텐츠(장르)만으로 추천하면 실제 사용자 취향 패턴을 반영하지 못합니다.
- 사용자 로그인/프로필 없이도 "영화 하나 선택 -> 추천" 흐름에 맞는 알고리즘이 필요합니다.

### 사용자에게 제공하는 가치

검색 한 번, 선택 한 번만으로 비슷한 영화 목록과 "왜 추천되었는지"에 대한 설명을 함께 제공합니다.

### 최종 서비스 기능

- 영화 제목 검색/선택
- 선택한 영화 기준 Top-N 유사 영화 추천 (기본 10개)
- 추천 이유(공통 장르 또는 협업 필터링 근거) 표시

## 11.2 추천 시스템 개념

- **추천 시스템**: 사용자가 좋아할 만한 아이템(영화, 상품 등)을 예측해 순위를 매기는 시스템. 일반적인
  회귀/분류가 "하나의 정답값"을 예측하는 것과 달리, 추천 시스템은 **"여러 후보 중 상위 몇 개를
  고르는 순위 문제(Ranking)"**에 가깝습니다.
- **사용자와 아이템**: 이 프로젝트에서 사용자는 `user_id`, 아이템은 영화(`movie_id`)입니다.
- **명시적 피드백 vs 암묵적 피드백**: MovieLens의 평점(1~5점)은 사용자가 직접 남긴 **명시적
  피드백**입니다. 클릭/시청 여부만 기록되는 암묵적 피드백과 달리, "얼마나 좋아하는지"의 정도를
  직접 알 수 있다는 장점이 있지만, 사용자가 평가를 남긴 영화만 알 수 있다는 한계(선택 편향)도
  있습니다.
- **평점 데이터의 의미**: 평점은 "본 영화 중에서" 매긴 것이므로, 평점이 없는 영화가 "싫어하는
  영화"라는 뜻은 아닙니다. 단지 "아직 평가하지 않은 영화"입니다.
- **개인화 추천과 비개인화 추천**: 인기 영화 Top 10처럼 모든 사용자에게 동일한 결과를 주는 것이
  비개인화 추천이고, 사용자/아이템 특성에 따라 결과가 달라지는 것이 개인화 추천입니다.
- **Cold Start 문제**: 평점 데이터가 거의 없는 신규 영화, 혹은 평가 이력이 없는 신규 사용자에게는
  협업 필터링이 제대로 동작하지 않습니다. 이 프로젝트는 신규 영화 문제를 콘텐츠 기반(장르) 추천으로
  보완합니다.

## 11.4 EDA

### 데이터 구조

`ratings`(100,000 x 4), `movies`(1,682 x 23, `video_release_date` 컬럼 제외 후), `users`(943 x 5).
`ratings`는 `movie_id`로 `movies`와, `user_id`로 `users`와 연결되는 관계 테이블입니다.

### 기술통계 / 결측치 / 중복치

- `ratings`에는 결측치와 중복 평가(동일 user_id+movie_id) 모두 없습니다.
- `movies`의 `release_date`는 1건 결측 - movie_id 267 `"unknown"` (실제 영화가 아닌 placeholder
  항목). `imdb_url`은 3건 결측. `video_release_date`는 전체가 결측이라 아예 사용하지 않습니다.
- **영화 제목 중복 18쌍(36편)** 발견 — 서로 다른 `movie_id`에 동일 제목이 등록된 MovieLens 100K의
  알려진 데이터 품질 이슈입니다. 두 ID에 각각 다른 평점이 쌓여 있어 임의로 병합하면 데이터 손실이
  발생하므로, **병합하지 않고 그대로 유지**했습니다(한계로 기록).

### 평점 분포

평균 3.53점, 중앙값 4점, 표준편차 1.13, **왜도 -0.51(음의 왜도)**. 4점이 34,174건으로 가장 많고,
3~5점이 전체의 약 83%를 차지합니다. 사용자들이 전반적으로 관대하게 평점을 매기는 경향이 있습니다.

### 사용자별 / 영화별 평점 수

- **영화별 평점 수**: 평균 59.45건, 중앙값 27건, **왜도 2.31(강한 양의 왜도)**. 141편은 평점이
  단 1건, 333편은 5건 미만, 743편은 20건 미만입니다. 전형적인 Long-Tail 분포입니다.
- **사용자별 평점 수**: 평균 106.04건, 중앙값 65건, **왜도 1.91**. 최소 20건(데이터셋 자체 필터링
  기준), 최대 737건.

### 영화별 평균 평점

가장 많이 평가된 영화는 *Star Wars (1977)* (583건, 평균 4.36점). 반대로 평점이 1~2건뿐이면서
평균이 5.0점인 영화도 다수 존재해, **"평균 평점만으로 순위를 매기면 안 되는"** 근거가 됩니다.

### 인기 영화 / 장르 분포

인기도(15.1절 Bayesian Weighted Rating) 기준 상위권은 *Schindler's List*, *Star Wars*,
*Shawshank Redemption* 등 잘 알려진 작품들입니다. 장르는 `Drama`(725편), `Comedy`(505편) 순으로
많고 `Fantasy`, `Film-Noir`, `unknown`은 매우 적습니다(장르당 영화 수는 8~14배 차이).

### 사용자 인구통계 / 개봉연도

사용자 평균 연령 34세(왜도 0.605, 완만한 오른쪽 치우침), 남성 670명 / 여성 273명. 개봉연도는
1993~1997년에 집중되어 있습니다(데이터 수집 시점인 1997~1998년 직전 개봉작이 많음).

### 데이터 품질 문제 요약

1. 영화 제목 중복 18쌍 (병합하지 않고 유지)
2. `movie_id=267 "unknown"` placeholder (개봉연도 없음, 장르 `unknown`만 보유)
3. Long-Tail 분포로 인한 Cold Start 대상 영화 743편(44%)

## 11.5 Feature Engineering

| Feature | 목적 | 적용 전후 변화 | 추천 품질 영향 |
|---|---|---|---|
| `rating_count`, `rating_mean` | 영화 신뢰도/품질 파악 | 원본에는 없던 집계 컬럼 추가 | 인기도 계산의 입력 |
| `popularity_score` (Bayesian Weighted Rating) | 평점 수와 평균을 함께 반영한 인기도 | 평점 1~2건짜리 5점 영화가 상위권에서 사라짐 | Cold Start용 기본 추천, `/movies` 기본 정렬 |
| 최소 평점 수 필터 (`MIN_RATING_COUNT_FOR_CF=20`) | 협업 필터링 신뢰도 확보 | 1,682편 -> 939편(56%)만 CF 대상 | CF 유사도의 통계적 안정성 확보 |
| `clean_title` / `release_year` / `search_title` | 검색 UX, 화면 표시 | 제목에서 연도 분리(예외 케이스 2건 안전 처리) | 유사도 계산에는 미사용, UX 전용 |
| `genre_matrix` (Multi-Hot) | 콘텐츠 기반 추천 입력 | 1,682 x 19 이진 행렬 | 장르 코사인 유사도 계산 |
| `user_item_matrix` (평균 중심화) | 협업 필터링 입력 | 943 x 939 희소 행렬(채움 비율 약 6.3%) | 아이템 간 코사인 유사도 계산 |

**주의사항**: 사용자-영화 평점 행렬의 결측값(미평가)은 "중립적인 평점"이 아니라 "정보 없음"이므로,
사용자 평균으로 중심화한 뒤에만 0으로 채웁니다. 이 근사는 평점 수가 매우 적은 영화/사용자에서
유사도를 왜곡할 수 있는 한계가 있어, 최소 평점 수 필터로 일부 완화합니다.

**데이터 누수 방지**: `release_year`, `clean_title` 등은 검색/표시 전용이며 유사도 계산에 사용하지
않습니다. 알고리즘 비교(11.7절) 시 유사도 행렬과 인기도 점수는 `ua.base`(학습 분할)만으로 계산하고,
평가 시 해당 사용자가 이미 `ua.base`에서 본 영화는 추천 후보에서 제외해 정보 누수를 방지했습니다.

## 11.6 추천 알고리즘

### Popularity-Based Recommendation

`weighted = (v/(v+m))·R + (m/(v+m))·C` (v=평점 수, R=영화 평균, m=최소 투표 기준, C=전체 평균).
**장점**: 계산 비용이 거의 없고 Cold Start에 강함. **단점**: 개인화가 전혀 없어 입력(선택한 영화)과
무관하게 항상 동일한 결과를 반환함.

### Content-Based Recommendation (장르)

영화의 장르 Multi-Hot 벡터 간 코사인 유사도. **장점**: 평점 데이터가 전혀 없어도 계산 가능(Cold
Start에 가장 강함), 결과 설명이 직관적("같은 장르라서"). **단점**: 실제 사용자 취향 패턴을 반영하지
못해 정량 성능이 가장 낮음.

### Item-Based Collaborative Filtering

사용자 평균으로 중심화한 평점 행렬에서, 영화(열) 벡터 간 코사인 유사도. **장점**: 실제 사용자 행동
패턴을 반영해 장르가 달라도 "함께 좋아하는" 영화를 찾아냄, 사용자 프로필 없이 "영화 한 편 선택"만
으로 계산 가능. **단점**: 평점 수가 부족한 영화(939/1,682편만 대상)에는 적용 불가.

## 11.7 모델 평가

### 평가 방법

MovieLens 100K 공식 분할(`ua.base` 90,570건 / `ua.test` 9,430건, 사용자당 정확히 10건)을 사용했습니다.
각 사용자에 대해 `ua.base`에서 가장 높게 평가한 영화 1편을 "선택 영화"로 삼고, 이를 기준으로 생성한
Top-10 추천이 `ua.test`에서 그 사용자가 4점 이상 평가한 영화("정답 집합")와 얼마나 겹치는지
Precision@10 / Recall@10으로 측정했습니다. 이는 "영화 한 편 선택 -> 추천"이라는 실제 서비스 흐름을
그대로 재현한 평가입니다. 데이터 분할 시 유사도 행렬/인기도는 `ua.base`만으로 계산해 시간 순서상의
정보 누수를 방지했습니다.

### 정량 비교 결과

| 알고리즘 | 평가 대상 사용자 | Precision@10 | Recall@10 |
|---|---|---|---|
| Popularity | 934 | 0.0468 | 0.0815 |
| Content-Based (장르) | 934 | 0.0102 | 0.0167 |
| Item-Based CF | 906 (전체의 97%) | 0.0417 | 0.0726 |

### 정성적 비교

| 기준 | Popularity | Content-Based | Item-Based CF |
|---|---|---|---|
| 개인화 수준 | 없음 | 있음(장르) | 있음(행동 패턴) |
| Cold Start(신규 영화) | 불가 | 강함 | 불가(20건 미만 제외) |
| 서비스 구조 적합성 | 낮음(입력 무관) | 있음 | 높음 |
| 계산 비용 | 매우 낮음 | 낮음 | 중간(사전 계산 필요) |

### 해석

Popularity가 두 지표 모두에서 Item-Based CF를 근소하게 앞섭니다 - "인기도는 매우 강력한
baseline"이라는 추천 시스템 연구에서 잘 알려진 현상입니다. 그러나 Popularity는 **선택한 영화와
무관하게 항상 같은 결과**를 반환하므로 이 서비스의 목적(영화 한 편 선택 -> 유사 영화 추천)에 맞지
않습니다. 지표의 절대값(0.01~0.08)은 "영화 한 편만으로 예측"하는 제한적 평가 조건 때문에 낮게
나오는 것이 자연스러우며, 절대값보다 알고리즘 간 상대적 비교와 서비스 구조 적합성이 더 중요한
판단 기준입니다.

## 11.8 최종 모델 선정

### 최종 알고리즘

**Item-Based Collaborative Filtering(주 알고리즘) + Content-Based 장르 유사도(Cold Start 대체)**

### 선정 이유

1. Popularity와 대등한 수준의 정량 성능을 보이면서도, 선택한 영화에 따라 실제로 다른 결과를 제공해
   서비스 목적에 부합합니다.
2. Content-Based보다 뚜렷하게 높은 정량 성능(Precision@10 기준 약 4배)을 보입니다.
3. 평점이 부족한 영화(전체의 44%)는 Content-Based로 자동 대체해 "추천 자체가 불가능한 상황"을
   방지합니다.
4. 두 방식 모두 사전에 계산된 유사도(행렬 조회 또는 소규모 장르 코사인 유사도)만 사용해 FastAPI에서
   실시간 응답이 가능합니다.
5. 두 점수를 억지로 가중합산하는 복잡한 하이브리드가 아니라, "주 알고리즘 -> 실패 시 대체
   알고리즘"이라는 단순한 조건 분기 구조를 사용합니다.

### 운영 서비스 적용 방식

`train_model.py`가 두 산출물(`similarity_matrix.pkl`, `movie_features.pkl`)과 메타데이터
(`movie_metadata.pkl`)를 저장하고, `main.py`가 서버 시작 시 이를 한 번만 로드해 `POST /recommend`
요청마다 재사용합니다.

### 한계점

- Item-Based CF는 평점 수가 경계값(20~50건) 근처인 영화에서는 소수 사용자의 평점에 유사도가 좌우될
  수 있습니다.
- Content-Based는 장르만 사용하므로 "장르는 다르지만 분위기가 비슷한 영화"는 찾지 못합니다.
- 정량 평가가 "영화 한 편만으로 예측"하는 제한적 조건이라, 전체 시청 이력 기반 추천 시스템과 절대적
  성능을 직접 비교할 수 없습니다.
- 중복 제목 18쌍을 병합하지 않아 검색 결과에 유사 항목이 중복 노출될 수 있습니다.

### 향후 개선 방향

- 여러 seed 영화(사용자가 여러 편을 선택)를 조합하는 세션 기반 추천으로 확장
- 최소 평점 수 기준을 데이터 증가에 따라 주기적으로 재산정
- 장르 외 배우/감독/줄거리 텍스트 등 추가 콘텐츠 Feature 도입 검토

## 11.9 서비스 구조

```
ai-server/fe/movie_recommendation_analysis.ipynb   EDA, Feature Engineering, 알고리즘 비교/선정
              │
              ▼ (선정된 알고리즘 반영)
ai-server/train_model.py                            데이터 로딩 → 정제 → FE → 유사도 행렬 생성 → 저장
              │
              ▼ (모델 산출물: similarity_matrix.pkl, movie_metadata.pkl, movie_features.pkl)
ai-server/main.py (FastAPI)                          산출물 로드 → /movies, /recommend API 제공
              │
              ▼ (REST API)
frontend/ (React + Vite)                             영화 검색/선택 → 추천 결과 화면 표시
              │
              ▼
docker-compose.yml                                   backend(8000) + frontend(5173) 통합 실행
```

Notebook은 연구/의사결정 기록, `train_model.py`는 그 결정을 반영한 운영용 산출물 생성, `main.py`는
산출물을 서비스로 노출, React는 사용자 인터페이스, Docker는 두 서비스를 하나의 명령으로 통합
실행하는 역할을 담당합니다.
